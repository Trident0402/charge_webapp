- 各銀行帳戶
- Line Pay Money
- 錢包
- 股票帳戶

- HTML
- CSS

- 後端
- React / Vue / Angular
- Node 打包工具
- 雲端同步
- 登入系統

## 2. 簡化架構原則

檔案不要拆太細。同一個主題放在同一個檔案裡，資料夾最多只保留一層。

例如：

- 帳戶相關：放 `js/accounts.js`
- 收入支出相關：放 `js/transactions.js`
- 股票相關：放 `js/stocks.js`
- 預期入帳相關：放 `js/expected-income.js`
- 設定與備份：放 `js/settings.js`

不要再拆成：

```text
features/accounts/accounts.logic.js
features/accounts/accounts.render.js
features/accounts/accounts.form.js
```

這樣對目前的小型 App 太複雜。

- 選單包含：

  - 賣出
  - 新增市價

- 從 `stockTrades` 中整理已輸入過的股票代號與股票名稱
- 下一次新增股票交易時，股票代號與股票名稱欄位可以從歷史清單選擇
- 選到股票代號時，自動帶入對應名稱
- 選到股票名稱時，自動帶入對應代號

股票持股統合：

- 同一股票代號相同，視為同一持股
- 若股票代號不同但股票名稱相同，也視為同一持股

- 平均成本 = 統合後總成本 / 統合後持股數
- 最新市價優先使用最新日期的市價紀錄
- 若沒有市價紀錄，使用最新一筆買入價格作為市價輔助

## 7. 計算邏輯

### 一般帳戶餘額

```text
餘額 = 初始金額 + 收入總額 - 支出總額 + 轉入總額 - 轉出總額
```

### 股票持股數

```text
持股數 = 買入股數總和 - 賣出股數總和
```

### 股票平均成本

```text
平均成本 = 目前總成本 / 目前持股數
```

### 股票市值

```text
股票市值 = 持股數 * 最新手動市價
```

如果沒有市價紀錄：

```text
最新市價 = 平均成本
```

### 股票未實現損益

```text
未實現損益 = 股票市值 - 總成本
```

### 股票報酬率

```text
報酬率 = 未實現損益 / 總成本 * 100
```

### 總資產

```text
總資產 = 一般帳戶餘額總和 + 股票帳戶市值總和
```

## 12. 近期修正計畫

### 12.1 紀錄修改與刪除

- 一般收支、轉帳、股票買賣、股票市價、預期入帳的列表紀錄，改成點擊整個紀錄方框進入修改頁。
- 列表上不另外放「修改 / 刪除」小按鈕，避免手機畫面太擠。
- 刪除動作放在各自的修改頁面底部，使用者進入該筆資料後再刪除。

### 12.2 股票持股收合顯示

- 持股區塊使用可展開 / 收合設計。
- 未展開時盡量壓縮成兩行，主要顯示股票識別、股數、未實現損益、損益率。
- 展開後顯示最新市價、均價、總市值、總成本。
- 展開內容不顯示估計稅費，避免畫面資訊過多。

### 12.3 未實現損益計算

- 每一檔持股的未實現損益需扣除預估賣出相關成本。
- 預估成本公式：

```text
預估扣除成本 = 目前市值 * (交易稅 0.3% + 券商手續費 0.1425% * 2)
```

- 持股未實現損益公式：

```text
未實現損益 = 總市值 - 總成本 - 預估扣除成本
```

- 股票帳戶上方的未實現損益，直接加總所有持股的未實現損益，不再用總市值減總成本重新計算。

### 12.4 餘額調整

- 一般帳戶右下角 `+` 選單移除「餘額調整」。
- 新增 / 修改收支表單不再提供「調整」作為可選類型。
- 若舊資料裡已存在調整紀錄，仍可透過點擊紀錄進入修改頁並刪除。

### 12.5 初始預設帳戶

- 第一次開啟 App、或清除資料後重新建立初始資料時，先建立常用資產帳戶。
- 預設帳戶如下：
  - 玉山銀行：銀行帳戶
  - 富邦銀行：銀行帳戶
  - Line pay money：Line Pay Money
  - 現金：錢包
  - 富邦證卷：證卷帳戶
- 預設初始金額皆為 `0`。
- 已經有 localStorage 舊資料的使用者，不自動覆蓋或新增這些帳戶，避免產生重複資料。

### 12.6 設定頁補齊基礎帳戶

- 設定頁新增「新增基礎帳戶」按鈕。
- 按下後檢查目前帳戶清單，補齊以下五個基礎帳戶：
  - 玉山銀行：銀行帳戶
  - 富邦銀行：銀行帳戶
  - Line pay money：Line Pay Money
  - 現金：錢包
  - 富邦證卷：證卷帳戶
- 若已存在相同名稱或相同預設 ID 的帳戶，就視為已存在，不再新增。
- 如果使用者已經有重複帳戶，系統不自動合併、不刪除、不更動既有資料。
- 執行後顯示本次新增幾個帳戶；若五個都已存在，提示已不需新增。

### 12.9 預期入帳全資料統計

- App 內預期入帳相關標題不再加上月份限定文字，統一改成「預期入帳」。
- 預期入帳總額不再只計算特定月份，改為統計所有預期入帳資料。
- 首頁「預期入帳總額」要顯示所有預期入帳的總額。
- 預期入帳頁摘要也要顯示所有預期入帳的總額、已入帳、尚未入帳與筆數。
- 新增、修改、刪除、標記已入帳後，首頁與預期入帳頁都要能反映最新統計。

### 12.10 預期入帳清單範圍與卡片密度

- 預期入帳頁的摘要仍統計所有預期入帳資料。
- 預期入帳清單只顯示連續兩個月資料：
  - 使用者選定月份
  - 選定月份的下一個月
- 月份選項放在清單區塊左上角，使用 `<input type="month">`。

  - `cryptoPrices`：每日市價紀錄。
- 虛擬貨幣交易欄位：
  - 買入 / 賣出
  - 幣種代號
  - 幣種名稱

  - 新增買入、賣出、新增市價都放在右下角 `+` 操作選單。
- 首頁統計：
  - 總資產加入虛擬貨幣台幣市值。
  - 首頁小統計新增虛擬貨幣市值。

### 12.17 月報表子頁選單與實現損益月曆

- 將月報表頂部「圓餅圖 / 月曆」切換改成與頁面內容連在一起的 tab 樣式。
- 圓餅圖子頁不再使用大型「支出 / 收入」segmented control，改成子頁內的簡約三條線下拉式選單。
- 月曆子頁也加入同樣的三條線下拉式選單。
- 圓餅圖子頁選單項目：
  - 支出
  - 收入
- 月曆子頁選單項目：
  - 收入支出
  - 收入
  - 支出
  - 股票實現損益
  - 虛擬貨幣實現損益
- 股票實現損益計算：
  - 依股票帳戶買賣紀錄排序。
  - 買入增加股數與台幣成本。
  - 賣出依目前平均成本計算成本基礎。
  - 實現損益 = 賣出金額 - 成本基礎 - 賣出手續費 - 賣出稅費。

  - 買入用美元價格與匯率換算台幣成本。
  - 賣出用美元價格與匯率換算台幣收入。
  - 實現損益 = 賣出台幣收入 - 成本基礎 - 手續費美元換算台幣。
  - 結果依賣出日期放入月曆。
- 月曆格子依選單模式顯示對應數值，損益可顯示正負值。

- 純利潤不再只是月報表的一筆實現收益。
- 純利潤也要增加該幣種持倉庫存。
- 純利潤欄位使用方式：
  - 數量：本次收到的幣種數量。
  - 美元金額 / 利潤：本次收到幣種的美元總價值。
  - 台幣美金匯率：將美元總價值換算為台幣。
  - 手續費：可扣除美元手續費。
- 純利潤加入持倉時：
  - 增加該幣種數量。
  - 以美元總價值換算台幣後加入該幣種成本。
  - 手續費只扣在月報表的實現收益裡，不降低庫存成本。
  - 若有數量，會用美元總價值 / 數量作為該筆參考單價。
- 純利潤仍會納入月報表「虛擬貨幣帳戶實現收益」。
- 這樣穩定幣收益可以同時反映在庫存與損益報表中。

### 12.35 銀行帳戶初始金額與轉帳手續費設定

- 銀行帳戶的「修改帳戶」頁面新增可修改初始金額。
- 目前只有銀行帳戶開放在修改頁調整初始金額；其他一般帳戶維持既有規則，避免誤改歷史餘額。
- 銀行帳戶的「修改帳戶」頁新增手續費設定開關。
- 手續費設定可獨立開啟或關閉。
- 開啟後可填寫兩種預設手續費：
  - 轉帳到其他銀行帳戶時的「轉帳手續費」。
  - 轉帳到錢包帳戶時的「提領手續費」。
- 使用轉帳功能時，若來源帳戶是銀行帳戶且已開啟手續費設定：
  - 轉帳到銀行帳戶：自動新增一筆支出，分類為「轉帳手續費」。
  - 轉帳到錢包帳戶：自動新增一筆支出，分類為「提領手續費」。
  - 手續費支出建立在轉出銀行帳戶中，因此會從該銀行帳戶扣款。
- 自動新增的手續費支出仍沿用一般支出資料格式。
- 自動新增的手續費支出可像一般支出一樣：
  - 點擊紀錄方框修改。
  - 在修改頁刪除。
  - 修改日期、金額、分類與備註。
- 轉帳手續費與提領手續費不影響轉入帳戶金額，只影響轉出銀行帳戶餘額。
- 修改或刪除轉帳紀錄時，需一併考慮自動產生的手續費紀錄：
  - 第一版可先讓手續費紀錄作為獨立支出存在，不強制跟轉帳紀錄同步刪除。
  - 若後續需要，可以再加入 `feeForTransferId` 關聯欄位，讓轉帳與手續費可同步更新或刪除。

### 12.36 支出表單可選轉帳手續費

- 一般支出表單新增「加入轉帳手續費」checkbox。
- 只有在交易類型為「支出」時顯示此 checkbox。
- 勾選「加入轉帳手續費」後，才顯示「轉帳手續費」金額欄位。
- 未勾選時，不顯示手續費金額欄位，也不新增手續費支出。
- 手續費金額欄位預先帶入目前帳戶設定的「轉帳手續費」。
- 若目前帳戶沒有設定轉帳手續費，預設為 0，使用者仍可自行輸入。
- 儲存支出時：
  - 原本支出照常建立。
  - 若有勾選且手續費金額大於 0，額外建立一筆一般支出。
  - 額外支出的分類為「轉帳手續費」。
  - 額外支出的日期與原支出相同。
  - 額外支出建立在同一個帳戶中，因此會從該帳戶扣款。
- 自動新增的轉帳手續費支出仍是一般支出紀錄，可以自由修改與刪除。
- 第一版只在新增支出時自動建立手續費；修改既有支出時不自動重建或同步手續費，避免覆蓋使用者已修改的獨立手續費紀錄。

### 12.37 支出手續費欄位位置調整

- 在月報表頁面中，於「月曆 / 圓餅圖」子標籤旁新增第三個子頁：「年度資產總覽」。
- 子頁順序調整為：

- 年份切換沿用月報表目前的日期狀態，或在年度資產總覽子頁上方提供簡潔的上一年 / 下一年切換。
- 每個月份需要統計並呈現：
  - 非工資收入。
  - 總收入。
  - 總支出。
  - 月損益。
  - 月總資產。
  - 負債。
- 指標定義：
  - 總收入：該月一般帳戶收入加總，不包含轉帳轉入、不包含負債借款、不包含預期入帳。
  - 非工資收入：總收入中排除分類為「薪資」的收入。
  - 總支出：該月一般帳戶支出加總，不包含轉帳轉出、不包含負債還款；手續費支出需納入。
  - 月損益：總收入 - 總支出。
  - 月總資產：以該月月底狀態估算現金類帳戶餘額 + 股票市值 + 虛擬貨幣市值。
  - 負債：以該月月底狀態估算未還負債。
- 股票與虛擬貨幣月總資產估算：
  - 優先使用該月最後一筆市價紀錄。
  - 若該月沒有市價紀錄，使用該月以前最新一筆市價紀錄。
  - 若仍沒有市價紀錄，使用持倉成本或最近交易價格作為保守估算。
- 負債月末估算：
  - 使用負債帳戶的借款總額與該月底以前的還款紀錄估算。
  - 未還負債 = 借款總額 - 該月底以前已還款金額，最低不小於 0。
- 圖形呈現方式：
  - 手機優先使用縱向 12 個月清單，每個月份是一個壓縮資訊列。
  - 每個月份顯示月份名稱與 2 到 3 個主要數字：月損益、月總資產、負債。
  - 每個月份用水平長條呈現收入 / 支出 / 損益關係。
  - 月總資產與負債可用第二條細長條或雙色小圖形呈現，避免單一卡片塞太多文字。
  - 點擊月份後可展開顯示完整六個指標。
- 顏色規劃：
  - 收入使用綠色系。
  - 支出使用紅色系。
  - 月損益正值使用綠色，負值使用紅色。
  - 月總資產使用深色或藍灰色。
  - 負債使用灰色系，延續目前負債頁的視覺語言。
- 互動設計：
  - 子頁頂部保留與月曆 / 圓餅圖一致的子標籤樣式。
  - 右側三條線選單可先不影響年度資產總覽；第一版年度總覽固定顯示上述六個指標。
  - 若之後需要，可再讓三條線選單控制顯示欄位。
- 實作檔案規劃：
  - `js/monthly-report.js`：新增年度資產總覽 state、統計函式、渲染函式與事件綁定。
  - `css/style.css`：新增年度總覽清單、月份條狀圖、展開區塊樣式。
  - `service-worker.js`：更新快取版本。
- 第一版不新增新的資料格式，只讀取既有帳戶、交易、股票、虛擬貨幣與負債資料做年度彙總。
- 第一版先以本地資料估算，不串接外部股價或匯率。

### 12.39 年度資產總覽的人性化展示設計

- 年度資產總覽的核心目標不是塞滿數字，而是讓使用者快速回答三個問題：
  - 今年哪幾個月賺得比較多？
  - 哪幾個月支出壓力比較大？
  - 資產與負債是往好的方向走，還是越來越緊？
- 第一眼只顯示「年度健康摘要」，避免一進頁面就是 12 個月份的數字牆。
- 年度健康摘要包含：
  - 年度累積損益。
  - 平均每月損益。
  - 年底總資產。
  - 年底負債。
  - 資產減負債後的淨值。
- 摘要文字要使用人性化語句，而不是只顯示標籤：
  - 例如：「今年目前累積結餘 $120,000」。
  - 例如：「平均每月多留下 $10,000」。
  - 例如：「負債比年初下降 $35,000」。
- 若數字為負，使用溫和但清楚的提示：
  - 例如：「今年目前多支出 $18,000」。
  - 例如：「這個月現金流偏緊」。
  - 避免使用過度警告或責備性的文字。
- 12 個月份不使用傳統表格，改用「月份故事列」。
- 每個月份故事列分成兩層：
  - 收合狀態：只顯示月份、月損益、月總資產、負債，並用小圖形呈現收入支出關係。
  - 展開狀態：顯示非工資收入、總收入、總支出、月損益、月總資產、負債完整數據。
- 收合狀態的設計重點：
  - 月份放左側，像時間軸一樣由 1 月到 12 月排列。
  - 中間用一條雙色水平條呈現收入與支出。
  - 右側顯示該月最重要的結果：月損益。
  - 月損益為正時顯示「留下」，為負時顯示「超支」。
- 展開狀態的設計重點：
  - 用 2 欄小型指標格呈現完整六個數字。
  - 把「非工資收入」獨立標出，讓使用者看見薪資以外的收入能力。
  - 把「月總資產」與「負債」放在同一列，方便比較財務壓力。
- 年度趨勢圖以簡單、可讀為主：
  - 使用一條資產走勢線。
  - 使用一條負債走勢線。
  - 若空間不足，手機上改成小型折線區塊，只顯示整體方向。
- 不追求精密金融圖表，而是追求「一眼看懂方向」：
  - 資產往上：視覺上呈現穩定上升。
  - 負債往下：視覺上呈現壓力下降。
  - 若資產下降且負債上升，使用簡短提醒文字。
- 非工資收入的展示方式：
  - 在月份展開內容中顯示。
  - 年度摘要可額外顯示「非工資收入占總收入比例」。
  - 若非工資收入連續數月增加，可顯示「非工資收入正在增加」。
- 對使用者最友善的排序不是數字大小，而是時間感：
  - 保持 1 月到 12 月的自然順序。
  - 不用排行榜取代年度時間線。
  - 若要顯示最佳 / 最差月份，只放在年度摘要中作為小提示。
- 顏色要輔助理解，不用過度鮮豔：
  - 收入：綠色。
  - 支出：紅色。
  - 月損益正值：綠色文字或小標籤。
  - 月損益負值：紅色文字或小標籤。
  - 資產：藍灰色或深色。
  - 負債：灰色。
- 數字呈現規則：
  - 大數字使用 `$120,000` 這類完整金額。

  - 月份列高度保持穩定，避免展開時整頁跳動過大。

  - 「支出較高」
  - 「負債下降」
  - 「資料不足」

- 同一畫面最多只出現一組主要導覽。
- 外層已經有 `月曆 / 圓餅圖 / 年度資產總覽` 時，年度總覽內不再放 tabs。

  - 外層子頁 tabs。
  - 年度內部 tabs。
  - 圖表模式 tabs。
- 最終結構變成：
  - 外層：選擇月報表子頁。

### 12.45 年度首頁三個入口交互重新校正

  - 若三個入口都無法點擊，代表問題可能不是單一卡片，而是年度首頁容器或頁面狀態切換本身。

  - 年度首頁上方的資產/負債指標也使用同一套 `data-annual-page` 交互。

  - `data-annual-page` 點擊事件已加入 `event.preventDefault()`。

### 12.46 2026 年度趨勢圖無法開啟根因修正

- 問題：只有 2026 年度無法點進「趨勢圖」。
- 判斷：
  - 2026 年度有實際收入/支出或資產資料，趨勢圖會產生金額短標籤。
  - 金額短標籤使用 `formatCompactCurrency()`，內部呼叫 `formatNumber()`。
  - `monthly-report.js` 原本沒有 import `formatNumber`，因此有資料的年度進入趨勢圖時會發生錯誤。
  - 沒有資料的年度因為不會產生部分數字標籤，所以看起來可以正常進入。
- 修正方向：
  - 從 `utils.js` 正確 import `formatNumber`。
  - 保留目前年度入口的點擊交互，不再重改卡片事件。
  - 更新 PWA 快取版本，避免手機仍使用舊版 JS。
- 實作紀錄：
  - `monthly-report.js` 已補上 `formatNumber` import。
  - PWA 快取版本更新為 `charge-app-v50`。
