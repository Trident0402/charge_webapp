# WebApp PDF 匯出實作總結

## 目的

這份文件整理了在一個 WebApp 中實作 PDF 匯出時：

- 必要流程是什麼
- 哪些做法容易出問題
- 這次實作過程中實際踩過哪些坑
- 最正確、最穩定的路線應該怎麼走

這份內容特別適合給未來要重寫 `pdf_exporter.js`、維護報表輸出、或做其他 WebApp PDF 功能的人參考。

---

## 一、WebApp 輸出 PDF 的必要流程

如果是前端 WebApp，沒有後端 PDF 服務，通常必要流程如下：

### 1. 先決定 PDF 的輸出策略

WebApp 輸出 PDF，大致有三種路線：

1. 直接把 HTML 丟給瀏覽器列印
2. 用 `html2canvas + jsPDF` 類型工具把頁面轉圖片再做 PDF
3. 用真正的 PDF 排版工具，自己畫文字、表格、圖形

對「有圖表、有中文、有既有 UI 風格」的專案來說，最常見的是第 2 種。

### 2. 建立一份專用的 PDF DOM

不要直接拿畫面上的 UI 去匯出。

正確做法是：

- 另外建立一個 `pdf-export-container`
- 在裡面渲染「專門給 PDF 用」的版面
- 讓它和 App 畫面分離

原因：

- 畫面 UI 常常有捲動、動畫、sticky、tab、響應式切版
- 這些都會干擾 PDF 截圖結果

### 3. 鎖定每一頁尺寸

PDF 是紙張，不是瀏覽器視窗。

一定要先把每頁尺寸固定，例如：

- A4 直式
- 固定頁寬
- 固定頁高
- 固定 padding

最穩定的做法是：

- 在 DOM 階段就把每頁鎖成固定像素尺寸
- 例如 A4 可先用 `794 x 1123 px` 當作 96 DPI 基準

### 4. 把內容切成一頁一頁

不要把整份長內容一次丟給工具自動切頁。

正確做法：

- DOM 上先切成很多個 `.pdf-page`
- 每個 `.pdf-page` 就是一頁
- 每頁內容自己控制，不依賴套件幫你猜

### 5. 等字體與圖表都完成

匯出前至少要確保：

- 字體已載入完成
- Chart.js 圖表已畫完
- DOM layout 已穩定

否則常見結果：

- 中文字型跳掉
- 文字換行不同
- 圖表空白
- 內容高度突然改變，導致分頁爆掉

### 6. 逐頁轉成 canvas

最穩的前端流程是：

1. 找到所有 `.pdf-page`
2. 每一頁 individually 用 `html2canvas`
3. 每個 canvas 再放入 PDF

這樣每頁都是獨立座標系，不容易出現整體偏移。

### 7. 用 PDF 工具逐頁輸出

例如：

1. 建立 `jsPDF`
2. 第一頁 `addImage`
3. 之後每頁 `addPage`
4. 再 `addImage`
5. 最後 `save`

這是前端報表匯出最可控的做法。

## 二、這次實作過程中，實際踩過的問題

下面是這次你反饋「有問題」時，對應到的技術問題。

### 問題 1：固定猜每頁 25 筆資料

這是最早版本最核心的問題之一。

當時做法是：

- 試算表固定 `25` 筆一頁
- 評語頁內容固定塞進第二頁

這種做法的問題：

- 不同字體載入後高度不同
- 中文換行會改變行高
- 評語/建議數量一變，高度就不同
- 備註字數一多，也會把表格撐高

結果：

- 被裁掉
- 頁尾被擠爆
- 有些內容跑到下一頁但切不乾淨

### 問題 2：整份內容交給 `html2pdf` 自動切頁

這是後來「重度左偏」最可疑的來源。

當時邏輯是：

- 把整份報告包成一個大容器
- 直接 `html2pdf().from(container).save()`
- 讓套件自己處理多頁切割

這種做法的風險：

- 整張長畫布非常大
- `html2canvas` 先 rasterize 成大圖
- `html2pdf` 再把大圖切成多頁

常見副作用：

- 某些頁面左邊被裁掉
- 右側大片空白
- 水平座標偏移
- 每頁的可視範圍不一致

你後面提供的截圖，現象就非常像這種問題。

### 問題 3：把 PDF 容器放到超遠左側

曾經有一版做法是把匯出 DOM 放到：

- `left: -99999px`

這個做法表面上是為了「不要讓使用者看到」。

但對某些截圖工具來說，會造成：

- 抓圖基準座標異常
- 畫布只截到部分可視區
- 左上角起點不穩

結果就是：

- 整頁像被切掉一部分
- 左偏或裁切

### 問題 4：使用 `mm` 作為 DOM 版面單位

在 CSS 裡直接用：

- `width: 210mm`
- `height: 297mm`

理論上很直覺，但實務上不一定穩。

原因：

- DOM 渲染本質還是螢幕像素
- `html2canvas` 最終也是 rasterize
- `mm -> px` 的換算可能受裝置 pixel ratio、瀏覽器縮放、工具內部換算影響

這容易造成：

- 寬度估算誤差
- 分頁高度微差
- 畫布大小與 PDF 映射不一致

### 問題 5：直接依賴 `window.jspdf` / `html2canvas` 全域變數

後來改成逐頁輸出時，曾經改成直接手動呼叫：

- `html2canvas(...)`
- `new jsPDF(...)`

但你的專案實際載的是：

- `html2pdf.bundle.min.js`

這個 bundle 一定有 `html2pdf`
但不一定保證：

- `window.jspdf`
- `window.jsPDF`
- `window.html2canvas`

都能用同樣名稱取到。

所以後來發生了：

- 版面問題剛修一半
- 又出現「無法產生」的錯誤

這不是排版問題，而是依賴方式有風險。

### 問題 6：畫面 DOM 與 PDF DOM 沒完全隔離時的干擾

如果 PDF DOM 和主畫面共用太多 UI 狀態，常見會受到：

- 手機版 layout
- tab 顯示/隱藏
- `overflow`
- `position: sticky`
- 外層 transform

影響。

所以 PDF DOM 一定要是「完全獨立」的乾淨報表結構。

## 三、需要注意的地方

### 1. PDF 輸出不是畫面截圖而已

它其實是：

- 先排版
- 再定頁
- 再渲染
- 再輸出

只把它當成「按一下存成 PDF」通常會踩坑。

### 2. 先控制頁面，再談樣式

順序應該是：

1. 先確保每頁尺寸正確
2. 再確保每頁內容不溢出
3. 再做字體、色彩、美化

如果順序反過來，很容易在漂亮的 UI 上輸出出錯。

### 3. 每頁內容必須有高度預算

每個頁面要有明確可用高度。

例如：

- header 高度
- body 可用高度
- footer 高度

所有內容都是在這個 body 裡分配。

### 4. 表格與長文字最容易出事

尤其是：

- 備註欄
- 中文長字串
- 評語條列
- 建議條列

這些都要考慮：

- 是否換行
- 是否省略
- 是否切到下一頁

### 5. 匯出前要等待兩次 reflow

常見穩定做法：

- `document.fonts.ready`
- `requestAnimationFrame`
- 再 `requestAnimationFrame`

這能降低因為渲染未完成導致的高度誤判。

### 6. 圖表一定要在 PDF DOM 裡重畫

不要直接拿畫面上的 Chart canvas 複製。

最好做法是：

- 在 PDF DOM 裡放專用 canvas
- 針對 PDF 尺寸重畫一次圖表

這樣畫面版和 PDF 版就不互相干擾。

### 7. 不要讓套件替你猜太多

如果你想要「版面剛好」：

- 不要讓套件自動切頁
- 不要讓套件自動算內容高度
- 不要讓套件自己從超長畫布切

越多事情交給套件猜，越容易出現難解的偏移問題。

## 四、最正確的路線

如果未來要在一個 WebApp 裡做「真正穩定」的 PDF 匯出，我建議走下面這條路。

### 路線 A：前端純瀏覽器方案

這是最適合目前這個專案的路線。

#### Step 1

建立專用 PDF container：

- 不共用主畫面 DOM
- 不受 tab / scroll / responsive layout 影響

#### Step 2

先把整份報表切成很多 `.pdf-page`：

- 每頁固定尺寸
- 每頁 header / body / footer 固定

#### Step 3

用程式自己做分頁：

- 評語頁動態量高度
- 表格頁逐列加入，超高就換新頁

#### Step 4

等字體、圖表、layout 都完成

#### Step 5

逐頁轉 canvas：

- 一頁一頁轉
- 不用整份長畫布

#### Step 6

逐頁塞進 PDF：

- `jsPDF.addImage`
- `addPage`
- `save`

這條路最穩，最容易排錯，也最能保證版面一致。

## 五、最推薦的實作策略

### 最推薦結論

對這個專案來說，最正確的方向是：

**專用 PDF DOM + 固定 A4 像素頁面 + 自己分頁 + 每頁單獨轉 canvas + 再組成 PDF**

這比下面幾種方式都穩：

- 直接匯出畫面 DOM
- 整份 HTML 一次丟給 html2pdf
- 用固定筆數猜測每頁內容
- 依賴套件自動切頁

---

## 六、如果要避免再次重踩的規則

未來如果再重構 PDF 功能，請優先遵守下面規則：

1. 不要再用固定 `25` 筆這種猜測式切頁。
2. 不要把整份長報告一次交給 `html2pdf` 自動切頁。
3. 不要把 PDF DOM 丟到超遠座標後再期待截圖工具穩定。
4. 不要在 CSS 階段過度依賴 `mm`，優先用固定像素頁面作為畫布基準。
5. 不要假設 bundle 一定暴露 `html2canvas` / `jsPDF` 全域名稱。
6. 不要用主畫面的 responsive layout 直接生成 PDF。
7. 不要在字體和圖表還沒穩定前就開始匯出。

---

## 七、簡化版最佳流程

如果之後要快速重寫，可以直接照下面順序：

1. 建立 `pdf-export-container`
2. 產生多個固定尺寸 `.pdf-page`
3. 用程式做動態分頁
4. 等字體載入完成
5. 重畫 PDF 專用圖表
6. 每頁 individually 轉 canvas
7. 每頁加入 `jsPDF`
8. `save()`

---

## 八、最後結論

這次遇到的問題，不是單一 CSS 問題，而是整個 PDF 匯出策略需要更嚴謹。

真正穩定的 PDF 匯出，不是：

- 把一份漂亮 HTML 直接存成 PDF

而是：

- 把 HTML 當成「報表畫布」
- 先設計每一頁
- 先做分頁控制
- 再逐頁輸出

如果目標是：

- 中文正常
- 圖表正常
- 每頁版面剛好
- 不左偏不裁切

那最穩的實作路線就是：

**自己掌控每一頁，不把切頁工作完全交給套件。**
